## Jobs-to-be-Done Analysis

Conduct a structured JTBD analysis to identify user jobs, pain points, and underserved needs. Output feeds into `/bootstrap-product` to ground the PRD in real user motivations.

## Instructions for the Agent

**IMPORTANT: Follow this exact sequence. Ask questions ONE AT A TIME and wait for responses before proceeding.**

### Pre-Check: Prior Research

Before asking questions, check if `docs/MARKET-RESEARCH.md` exists.

**If it exists:**
1. Read the file
2. Tell the user: "I found your market research. I'll use the competitor analysis and audience insights as starting points."
3. Skip Question 1 — use the Product Concept from the market research instead
4. When analyzing current solutions and workarounds, cross-reference competitor weaknesses from the market research as candidate pain points

**If it does not exist:**
- Proceed with all questions. Mention that running `/research` first would enrich this analysis, but it is not required.

---

## Questionnaire

**Question 1** *(skip if market research exists)*
```
What product or idea are you analyzing?

Describe it in one sentence.

Example: "A fitness studio management platform for boutique gym owners"
```

**Question 2**
```
What is the desired outcome your users are trying to achieve?

Think beyond your product — what end result do they want in their life or work?

Example: "Fill every class slot without spending time on marketing"
Example: "Ship database changes without fear of breaking production"

Desired outcome:
```

**Question 3**
```
What situation triggers the need for this outcome?

When does the user realize they have this problem?

Example: "When they see empty class slots and declining revenue"
Example: "When a migration fails in staging and blocks the release"

Trigger situation:
```

**Question 4**
```
What is the main functional job the user is trying to get done?

(A functional job is a practical task — what they need to accomplish)

Example: "Manage class schedules and automatically fill open slots"
Example: "Test database migrations against production-like data"

Functional job:
```

**Question 5**
```
What emotional or social jobs are involved?

(How does the user want to feel? How do they want to be perceived?)

1. Feel confident / reduce anxiety
2. Look competent to peers or bosses
3. Save time for things that matter more
4. Feel in control of a chaotic process
5. Other (please describe)

Enter a number (1-5) or describe:
```

**Question 6**
```
What are the biggest frustrations with how people solve this today?

List 2-4 specific frustrations.

Example:
- "Have to manually text clients to fill cancelled slots"
- "No way to see which time slots perform best"
- "Existing tools are too expensive for a single-location studio"

Your frustrations:
```

## After Collecting All Responses

### Phase 1: Construct Core Job Statement

Build the core job statement using the JTBD format:

```
When [trigger situation from Q3],
I want to [functional job from Q4],
so I can [desired outcome from Q2].
```

### Phase 2: Optional Web Validation

If `WebSearch` is available, run validation searches to corroborate user-reported pain points:

**Validation searches (parallel):**
- `"{target audience} frustrations with {current solutions}"` — Reddit, forums
- `"{competitor} complaints reviews"` for known competitors
- `"{functional job} challenges {industry}"` — industry pain points

**IMPORTANT:** User answers are the primary source. Web research validates and enriches but never replaces what the user said. If web results contradict user input, note both perspectives.

**If `WebSearch` is unavailable:** Skip web validation. Mark all findings as "User-reported" in the output.

### Phase 3: Synthesize

Build the full JTBD analysis:

1. **Core Job Statement** — from Phase 1
2. **Functional Jobs** — break Q4 into 3-5 discrete sub-jobs
3. **Emotional/Social Jobs** — from Q5, expanded
4. **Current Solutions** — from Q6 and market research competitor data
5. **Pain Points** — merge Q6 frustrations with any web-validated findings
6. **Underserved Needs** — gaps where current solutions fail at the jobs
7. **Desired Outcomes** — using Ulwick's outcome-driven format: "Minimize the time it takes to [job step]"

### Phase 4: Write Output

Present a summary to the user and confirm before writing the file.

Write the output to **`docs/JOBS-TO-BE-DONE.md`** using the template below.

## What Happens Next

After completing JTBD analysis:
1. Run `/positioning` to define product positioning (uses both market research and JTBD as input)
2. Run `/bootstrap-product` to create the PRD (reads all three artifacts and pre-populates questions)
